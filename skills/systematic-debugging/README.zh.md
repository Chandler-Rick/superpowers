# 系统化调试

## 概述

随意修复只会浪费时间并制造新 bug。快速补丁掩盖了根本问题。

**核心原则：** 必须先找到根因，才能尝试修复。只修复表象等于失败。

**违反流程即违反调试精神。**

## 铁律

```
未查明根因，禁止修复！
```

如未完成第一阶段，禁止提出修复建议。

## 适用场景

适用于任何技术问题：
- 测试失败
- 生产 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其适用：**
- 时间紧迫时（越急越不能乱修）
- “只需一个小修复”时
- 已多次尝试修复未果
- 上次修复无效
- 问题未完全理解

**不要跳过：**
- 问题看似简单
- 急于求成
- 管理催促
