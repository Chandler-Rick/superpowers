# 编写计划

## 概述

编写详尽的实现计划，假设工程师对代码库和工具一无所知。记录每步涉及的文件、代码、测试、文档及测试方法。将整个计划拆分为小任务。遵循 DRY、YAGNI、TDD、频繁提交。

假设执行者技术熟练，但对本工具和领域不熟，对测试设计也不擅长。

**开始时声明：**“我正在使用 writing-plans 技能制定实现计划。”

**建议在头脑风暴技能创建的专用 worktree 中运行。**

**计划保存至：** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`
- （如有用户偏好，优先）

## 范围检查

如规范涉及多个独立子系统，应在头脑风暴阶段拆分为多个子项目规范。否则建议拆分，每个计划产出可独立测试的软件。

## 文件结构

在定义任务前，先规划将创建或修改的文件及其职责。此处决定模块划分。

- 设计单元需边界清晰、接口明确。每个文件只负责一件事。
- 只处理能一次性掌握的代码，文件越专注越好。小文件优于大文件。
- 相关变更文件应放一起。按职责分，不按技术层分。
- 现有项目遵循既有模式。若文件过大，可在计划中拆分。

结构决定任务拆分。每步应产出自洽的变更。

## 小步任务粒度

**每步只做一件事（2-5分钟）：**
- “写失败测试”——一步
- “运行确保失败”——一步
- “实现最小通过代码”——一步
- “运行测试确保通过”——一步
- “提交”——一步

## 计划文档头部

**每个计划必须以如下头部开头：**

```markdown
# [功能名] 实现计划
```

（后续内容可补充）
