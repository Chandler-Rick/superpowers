# Superpowers 中文项目使用说明与学习路线

这个目录是基于原项目 `README.md` 整理出的中文化入口，适合先快速理解项目，再回到原仓库逐步深入。

## 这个项目怎么用

Superpowers 本身不是传统意义上的业务应用，也不是一个执行某个单一功能的程序。它更像是一套“给 AI 编码代理加工作流”的技能系统。你安装之后，真正的使用方式是在和编码代理对话时触发这些技能。

最常见的使用过程是：

1. 先把 Superpowers 安装到你正在使用的代理平台，例如 Claude Code、Cursor、Codex、OpenCode 或 Gemini CLI。
2. 开一个新的代理会话，不要一上来就让它直接改代码，而是先给它一个清晰的目标，例如“帮我设计这个功能”“帮我排查这个 bug”“帮我把这个想法变成实现计划”。
3. 观察代理是否自动进入对应流程，比如先澄清需求、再输出设计、再拆计划、再按计划执行。
4. 在关键节点进行确认，尤其是设计确认、计划确认、代码审查和最终验证。
5. 把它当作一个工程流程增强器，而不是简单代码生成器。

## 适合什么场景

- 你想让代理先想清楚再写代码，而不是直接开始乱改。
- 你想把需求澄清、设计、计划、实现、测试、评审串成一个稳定流程。
- 你希望代理按测试驱动开发或至少按可验证的方式推进。
- 你要处理多步骤、跨文件、需要中间确认的软件开发任务。

## 不适合怎么用

- 不适合把它当成一个“安装后立刻出现 UI 页面”的桌面应用。
- 不适合期待它脱离代理平台独立运行。
- 不适合在没有项目上下文、没有需求目标的情况下让它凭空产出高质量工程结果。

## 从 0 开始学习这个项目

如果你之前完全没接触过这类代理工作流，建议按下面顺序学。

### 第 1 阶段：先理解它解决什么问题

先读这个目录下的 `README.md`，重点理解三件事：

- 它不是普通软件，而是一套给代理用的技能系统。
- 它的目标不是“更快写代码”，而是“更系统地完成软件开发”。
- 它强调设计、计划、测试、评审和验证，而不只是生成代码。

这一阶段的目标不是记住命令，而是建立正确预期。

### 第 2 阶段：理解核心工作流

重点掌握下面七个环节：

1. brainstorming
2. using-git-worktrees
3. writing-plans
4. subagent-driven-development / executing-plans
5. test-driven-development
6. requesting-code-review
7. finishing-a-development-branch

你可以把它理解成一条流水线：

需求澄清 -> 设计确认 -> 实现计划 -> 编码执行 -> 测试验证 -> 代码评审 -> 收尾交付

### 第 3 阶段：选一个平台真正装起来

建议只选一个平台先学，不要同时学多个。

- 如果你已经在用 Cursor，就先从 Cursor 开始。
- 如果你在用 Codex，就走 Codex 的安装方式。
- 如果你主要用 Claude Code，就直接走官方市场或插件市场安装。

目标是先把一次完整流程跑通，而不是一开始就覆盖所有平台。

### 第 4 阶段：做一个很小的真实练习

不要一开始拿复杂项目试。建议用一个小任务练手，比如：

- 给现有页面加一个按钮
- 修一个明确可复现的 bug
- 给接口补一个参数校验
- 把一个小功能拆成设计和实现计划

你需要重点观察：

- 代理有没有先问清需求
- 有没有输出可确认的设计
- 有没有把任务拆成具体步骤
- 有没有在实现前后做验证

### 第 5 阶段：阅读技能目录

当你已经体验过一次流程后，再回到原仓库看 `skills/` 目录。这时候你会更容易理解每个技能存在的原因，而不是只看到一堆说明文件。

推荐优先阅读：

1. `skills/using-superpowers/SKILL.md`
2. `skills/brainstorming/SKILL.md`
3. `skills/writing-plans/SKILL.md`
4. `skills/test-driven-development/SKILL.md`
5. `skills/requesting-code-review/SKILL.md`

### 第 6 阶段：理解它背后的工程思想

如果你想真正学会这个项目，而不是只会安装，应该重点吸收下面几种思想：

- 先澄清目标，再动手实现
- 复杂任务先拆分，再执行
- 实现必须可验证，不能只靠“我觉得没问题”
- 代码评审应该在流程中，而不是最后临时做
- 测试和验证是开发的一部分，不是收尾补丁

## 推荐学习方式

- 第一遍：只读中文 README，建立整体认知。
- 第二遍：挑一个平台安装并试一次小任务。
- 第三遍：回头对照原仓库里的 `skills/` 目录读技能文件。
- 第四遍：把它用在一个真实但小规模的项目里。
- 第五遍：再总结哪些技能最适合你的工作流。
